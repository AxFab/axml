xOS is a hobbyist operating system project written mostly from scratch for the PC, entirely in FASM-style assembly. The principal goal of xOS is to be compact and lightweight, yet be suitable for use in the 21st century, in terms of speed, features and hardware support.

Most of the important things in the world have been accomplished by people who have kept on trying when there seemed to be no hope at all.

​

Features

  • Fast. xOS can start in less than 5 seconds on a Pentium 4 with 512 MB of RAM.
  • Small. The binaries without data (such as wallpapers) are less than 1 MB.
  • Lightweight. A booted system uses less than 32 MB of RAM.
  • Graphical. xOS includes a compositing windowed graphical user interface.
  • Common hardware. xOS supports VESA BIOSes, PS/2, ACPI, PCI, ATA & SATA hard disks.
  • Self-hosting. xOS is capable of building itself under itself.
  • USB. xOS currently supports USB 1.1 with keyboards and mice.
  • Networking. xOS features a somewhat functional TCP/IP stack and a web browser.

Take a look

I commit regularly to the GitHub repository. Every time I commit, the file disk.hdd in the repository is updated, which can be considered the latest unstable build. Please give feedback using GitHub issues or by emailing me at omarx024@gmail.com.

Running xOS

Running xOS is very simple. Whether you have the latest unstable build or a release build, you will have an HDD file, which can be attached to the IDE or SATA controller in VirtualBox, where xOS runs best. For networking in VirtualBox, open the network settings page, set the adapter type to any one of the Intel Pro/1000 options, and set the "attached to" field to NAT. xOS needs about 32 MB of RAM to boot, while over 64 MB is recommended for networking.

Original kernel

SmokeOS incude it's own kernel. This means it's not yet another Linux distribution. The design is entierly original even if still close to Unix philosophy. For interoperability sake, I tried to include some of NT design (Windows).

Standard C library

A doubt that trouble me for quitte some time is the standard library. Should I use an exising one or create my own. As I needed to implement some of the feature for my kernel anyway I decided to give it a try. This gave me more freedom to design the kernel ABI. For the time being, I'm focusing on one standard version at a time. I just completed StdC89, and starting StdC95.

Virtual File-system

While still under heavy refactoring, the design I choose for the VFS is really closed to Linux. I tried as much as possible to simplify driver development, especially testing. Did I succeeded, only other developers might tell...

Frame manager

One area where there was much to do (still do) is the windows manager. Unix machine use mostly the X11 deamon, while proprietary system prefer to handle most of the drawing. I wanted to avoid re-routing of the messages while still let freedom to the frame applications. Sadly, none of the existing API suit me enough to achieve this goal. So I went to a solution inspired of Linux frame-buffers. However instead of mapping only physical display, my kernel can allocate one frame-buffer for each window. All the logic of the manager being keeped on the kernel, allowing fast and easy event dispatch.

And many more...

My OS is still a young pup and I need more time to try some of the countless ideas I have. That's all the fun to create a raw system, everything looks possible... My first objective right now is fixing some issues with signals and multi-threading (stable context-switchng is not that easy), starting my network stack and take a glance at ARM porting.
